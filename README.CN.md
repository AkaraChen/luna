# Luna

> 项目正在积极开发中。

**让你的 GitHub Project 积压工作自主完成。**

[English Version](./README.md)

---

## 为什么要做 Luna？

你有一个 backlog。那些 "Todo" 状态的 Issue 已经躺了好几周——不是因为它们太难，而是一天只有那么多小时。你需要帮手，但你不想盯着一个脚本看，也不想一步一步地哄着 AI 干活。

**Luna 基于一个简单的信念：编码 Agent 应该像一个优秀的异步队友那样工作——拿起任务、完成代码、提交 PR、继续下一个，不需要被催促。**

## 设计哲学

### 你的 backlog 本身就是真实来源
Luna 直接集成 GitHub Projects。不需要新的数据库，不需要迁移，不需要"把任务导入到这里"。你现在用的 Status 字段驱动一切。把卡片移到"In Progress"，Luna 就感知到了。移到"Done"，Luna 就停下来。

### 自主运行意味着持续运行
Luna 是一个长期运行的守护进程，不是一次性命令。它按配置的间隔轮询你的项目，将 Agent 分发到全新的工作区，监测卡死情况，失败时重试，并持续将运行中的工作与你的看板实际状态进行对账——全程不需要你盯着。

### 一个文件管理整个工作流
你的全部自动化逻辑都在一个 `WORKFLOW.md` 中。YAML 前置配置管理基础设施——并发限制、重试策略、权限配置、生命周期 Hook。正文是一个 Jinja2 模板，会成为 Agent 的提示词。修改文件后，Luna 无需重启即可生效。

### 隔离胜于聪明
每个 Issue 都有自己独立的工作区——Agent 从零开始在其中工作的全新目录。没有共享状态，并发任务之间不会相互干扰。工作完成后，工作区自动清理。

## 你可以做什么

**设好就不管**
- 指定一个 WORKFLOW.md 运行 `luna`，然后去做别的事
- Agent 自动按优先级顺序处理 Todo 和 In Progress 状态的任务
- 已完成的任务自动跳过；取消的任务立即停止对应的 Agent

**并发运行多个 Agent**
- 配置全局和按状态的并发数限制
- Agent 并行启动，每个都在独立的工作区中运行
- 速率限制和卡死检测防止费用失控

**精细控制 Agent 行为**
- 直接在 WORKFLOW.md 正文中编写 Agent 提示词——通过模板变量注入 Issue 标题、描述、URL、优先级、以及阻塞关系
- 选择权限预设（`high_trust`、`workspace_write`、`read_only`），或直接精细配置沙箱和审批策略
- 通过生命周期 Hook 在工作区创建后、每次运行前后、以及清理时执行 Shell 命令

**优雅地处理失败**
- Agent 失败或超时时，使用指数退避进行重试
- 卡死检测会终止无响应的 Agent 并重新调度
- 启动时自动清理上次遗留的过期工作区

## 适合谁用

Luna 适合以下开发者：
- 使用 GitHub Projects 管理工作
- 正在使用或探索自主编码 Agent
- 想要生产级的守护进程，而不是周末随手写的脚本
- 宁愿写一张清晰的任务卡，也不想全程盯着代码生成过程

## 快速开始

```bash
# 在当前目录初始化 WORKFLOW.md
luna init

# 在当前 issue 工作区里发布 tracker comment
luna comment "开始实现，接下来会补测试。"

# 启动编排器
luna
```

Luna 需要安装并完成认证的 [Codex](https://github.com/openai/codex) 和 [GitHub CLI](https://cli.github.com/)（`gh`）。

## 愿景

软件积压存在的根本原因是时间不够用。Luna 是迈向未来的第一步——在那个未来，backlog 是一个实时队列：需求以 Issue 的形式流入，Agent 以可用算力的形式接入，可运行的软件从另一端流出。人类留在关键决策层：决定做什么、审阅做出来的成果。
